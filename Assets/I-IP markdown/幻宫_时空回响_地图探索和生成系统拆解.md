**二、 随机地图生成系统（精简版）**

**2.1 生成原则**

-   **组合式生成**：每张地图由多个"区块"（Chunk）拼接而成，区块库对应地图主题。

-   **可控随机**：种子驱动，保证单局内一致性，多周目变化。

-   **专注核心循环**：区块设计聚焦于"战斗-收集-探索"，移除独立的事件系统。

**2.2 区块类型与配置（移除事件区块后）**

  ----------------------------------------------------------------------------------
  区块类型       出现概率    内容物                     设计意图
  -------------- ----------- -------------------------- ----------------------------
  **资源区块**   50%         密集材料点、少量普通敌人   提供稳定的资源获取

  **战斗区块**   40%         精英敌人、宝箱、地形挑战   提供核心战斗体验与主要掉落

  **安全区块**   10%         营火、无敌人               提供恢复与规划点
  ----------------------------------------------------------------------------------

  -------------- ------------ ------------ -----------------------------------

  **材料密度**   高           中           资源获取节奏

**2.4 可达性验证算法**

为确保每次生成的地图均可通关，在 `GenerateGridMap` 中集成可达性验证：

- **算法选择**：BFS（广度优先搜索），从安全区（营火）出发，在5x5网格中搜索至BOSS房间的路径
- **遍历规则**：仅将"可行走区块"（资源/战斗/安全类型）纳入搜索，不可行走区块（如墙壁、深渊）视为障碍
- **不连通处理**：
  1. 第一优先级：使用最小生成树（Kruskal算法简化版）找到两个连通分量之间最近的可行走节点对，添加"桥接路径"（将中间不可行走节点替换为资源区块）
  2. 第二优先级：若补全后仍不连通，递归重新生成整个网格（最多重试3次）
  3. 第三优先级：若3次重试均失败，强制在起点与目标之间生成一条直连路径（直线最短路径）

- **伪代码实现**：见第十章 `ValidateReachability` / `ConnectDisconnectedComponents`

**2.5 死胡同处理策略**

死胡同（Dead End）指只有单一入口且无法通向其他区块的末端区块，是玩家探索体验的常见痛点。

- **死胡同分级**：
  - 深度1（末端区块）：入口即死路，常见于资源区块
  - 深度2（次级末端）：进入后仅1个分支且分支为死胡同
  - 深度3+（深层死胡同）：多层级嵌套死路，探索成本高但回报感极低

- **处理策略**：
  - **深度1-2的死胡同**：30%概率添加宝箱（显式奖励），20%概率添加隐藏路径（通往附近资源区块），50%保持原样（作为"清场后回头"的正常探索节奏）
  - **深度>2的死胡同（强制处理）**：**100%添加奖励** —— 优先添加宝箱/资源簇（75%），若地形不允许则添加隐藏路径连接至主干道（25%），将"回头路"转化为"发现奖励"的正向体验
  - **隐藏路径视觉提示**：灵魂微光会向隐藏路径方向飘散，提供隐性引导

- **设计哲学**：深度>2的回头路是对玩家探索时间的"负投资"，必须转化为奖励事件。深度1-2的适度回头路有助于玩家建立"心智地图"的空间认知。

**2.6 节奏保护规则**

确保单局体验张弛有度，避免连续高强度战斗导致的疲劳或连续低强度导致的平淡。

- **核心规则**：**连续2个战斗区块后，下一个区块强制为资源类型或安全类型**
  - 若战斗区块A → 战斗区块B → 下一个必须是资源/安全
  - 例外：精英区块（BOSS前室）不计入连续计数，因为精英区块本身即为"高潮节点"，之后自然需要缓冲

- **区块序列生成修正算法**：
  ```
  consecutiveBattleCount = 0
  FOR EACH block IN generatedBlocks (按探索顺序):
      IF block.type == Battle:
          consecutiveBattleCount++
          IF consecutiveBattleCount >= 2:
              ForceNextBlockType = Resource OR Safe
      ELSE:
          consecutiveBattleCount = 0  // 非战斗区块重置计数
  ```

- **节奏基准**（期望值）：
  - 战斗区块之间的最大连续数：2
  - 安全区块最小间隔：每5-8个区块至少出现1次
  - 精英区块前后各预留1个非战斗区块作为"准备/恢复"窗口

1.  **环境互动（强化以替代事件功能）**：

        |------|------|------|

        | **飘散速度-垂直** | 0.05-0.15 m/s（上浮） | 轻微上浮，营造"灵魂升腾"视觉 |

        **颜色设计逻辑**：
        - `#4A3A8C`（深紫蓝）是游戏"灵魂"属性的主色调延续，与灵魂微尘材料、灵魂法术特效的视觉语言一致
        - 发光层`#7B6BA8`提供柔和的紫色光晕，与森林/荒漠等不同地图主题的背景色均能形成足够对比度（对比度比 >= 3.0，满足WCAG AA标准）
        - 隐藏路径方向粒子饱和度偏高(+10%)，提供"隐性"引导——玩家可能不会意识到的潜意识暗示
        - 颜色范围(Hue 250-260)的抖动避免粒子看起来像"复制粘贴"的机械生成，增强自然感

        | 难度 | 粒子密度 | 偏置强度 | 可见距离 | 设计意图 |
        |------|---------|---------|---------|---------|
        | 简单 | 18-25个/秒 | 1.5x | 15格 | 更明显的引导，降低探索挫败感 |
        | 普通 | 8-15个/秒 | 1.0x | 12格 | 标准参数 |
        | 困难 | 5-10个/秒 | 0.7x | 10格 | 更隐蔽的引导，需要更仔细观察 |
        | 疯狂 | 3-6个/秒 | 0.5x | 8格 | 极隐蔽，仅作为"最后手段"的提示 |

        **预期表现清单**：
        - [ ] 粒子颜色与灵魂微尘材料(#4A3A8C系)视觉一致
        - [ ] 隐藏路径方向粒子密度明显高于其他方向(1.5x)
        - [ ] 隐藏路径方向粒子形状为菱形(区别于圆形默认粒子)，提供额外视觉线索

        - [ ] 粒子飘散方向有向隐藏路径的水平偏置，形成"被风吹向秘密"的视觉暗示
        - [ ] 不同难度模式下粒子参数有明显差异，反馈在探索体验中
        - [ ] 粒子不遮挡敌人/材料/交互点的视觉识别
        - [ ] 粒子不受战斗状态影响，始终在隐藏路径入口附近生成

    -   **晶能流动**：指示区域能量流向，可能指向时空裂隙或隐藏敌人。

    -   **可破坏场景**：击碎特定晶簇可能掉落材料或触发敌人伏击。

    -   **记忆回响**：在记忆碎片区域自动触发莎娜记忆片段（剧情推进功能保留）。

**四、 连接机制详解**

**4.1 时空门系统**

-   **解锁条件**：

    1.  击败该区域的"时空守护者"（BOSS）

    2.  或携带对应的"记忆碎片"

-   **连接规则**：

    -   每个时空门连接2-3个特定区域

    -   连接关系部分随机，保证可访问所有区域

    -   已解锁的门可在营火处快速传送

**4.2 时空裂隙机制（P1-004 / P1-006 修复）**

**4.2.1 生成触发与概率叠加机制（P1-006）**

时空裂隙采用**加法叠加 + 硬上限封顶**的概率模型，避免未来扩展溢出：

-   **基础概率**：5%

-   **概率叠加因子（全部采用加法叠加）**：

    | 因子 | 概率加成 | 触发条件 |
    |------|---------|---------|
    | 中等负担 | +10% | 灵魂负担 > 50 |
    | 高负担 | +10%（与中等负担叠加） | 灵魂负担 > 70 |
    | 记忆碎片 | +3%/碎片 | 每个携带的记忆碎片 |
    | 多周目 | +3% | 周目 >= 2 |

-   **叠加公式**：
    ```
    裂隙出现概率 = MIN( 5%
                      + (burden > 50 ? 10% : 0%)
                      + (burden > 70 ? 10% : 0%)
                      + (memoryFragmentCount × 3%)
                      + (currentCycle >= 2 ? 3% : 0%),
                      50% )   // 硬上限
    ```

-   **概率上限（Cap）**：**最终概率硬上限 = 50%**。即使在极端叠满状态下（负担>70、7个记忆碎片、周目2+ = 5%+10%+10%+21%+3%=49%，略低于50%），也不得超过50%。

-   **未来扩展保护**：若后续版本新增概率加成因子（如新道具、新技能），继续使用加法叠加，但上限始终锁定50%。这确保裂隙出现频率不会泛滥，保持"偶然惊喜"的体验定位。

**4.2.2 "2选1"目的地预览与选择机制（P1-004）**

时空裂隙出现时，玩家不再被完全随机传送，而是获得短暂的"目的地预感"窗口，可二选一。

-   **预览触发**：时空裂隙生成后，在裂隙上方浮现两个半透明目的地标签，分别显示两个可能的目的地信息：

    | 标签信息 | 说明 |
    |---------|------|
    | 目的地名称 | 地图类型中文名（如"冰原""火山地""遗忘庄园""记忆碎片区域"等） |
    | 难度提示 | 彩色标记——绿色（同当前难度）、橙色（+1级难度）、红色（+2级难度） |
    | 预览奖励 | 简短奖励关键词（如"稀有材料""记忆碎片""精英灵魂之核"） |

-   **目的地生成规则**：
    -   从目的地图池中随机抽取2个不同的目的地
    -   目的地图池与原始裂隙规则一致：80%高难地图 + 20%记忆区域
    -   不排除当前所在地图类型（允许"同类型但不同实例"的传送）

-   **选择时限**：标签显示后，玩家有 **5秒** 的时间窗口通过以下方式做出选择：
    -   **键盘**：按 `A`/`D` 或 `←`/`→` 切换高亮选中项，按 `E` 或 `空格` 确认
    -   **鼠标**：直接点击目标标签确认
    -   **倒计时UI**：屏幕中央显示5秒倒计时条，随时间缩小

-   **超时处理**：若5秒内未做出选择，系统随机选择其中一个目的地（等概率50%/50%）。

-   **取消机制**：玩家可按 `Esc` 或离开裂隙交互范围（>3单位距离）取消选择，裂隙保持存在但不传送（可稍后重新靠近触发预览）。

-   **视觉表现**：
    -   两个标签分别漂浮在裂隙左右上方，带有灵魂微光拖尾效果
    -   当前高亮选中项放大110%并发出脉冲光晕
    -   倒计时条随剩余时间从绿色→黄色→红色渐变（5s绿→3s黄→1s红）

-   **设计意图**：
    -   **策略选择权**：将完全随机传送改为有限选择，让玩家根据当前资源/负担/目标做出战术决策
    -   **信息透明度**：预览机制降低"传送后后悔"的挫败感，同时保留目的地的不确定性
    -   **紧张节奏**：5秒限制制造"快速决策"的紧张感，与裂隙"不稳定通道"的叙事设定一致
    -   **不中断战斗流**：预览界面为轻量HUD叠加层，不暂停游戏，保持实时动作体验

-   **目的地类型细化**：

  -------------------------------------------------------------------------
  类型           概率        难度           奖励                           预览关键词
  -------------- ----------- -------------- ------------------------------ --------------------
  **高难地图**   80%         +1级（橙色）   稀有材料、精英灵魂之核         稀有材料 / 精英之核

  **记忆区域**   20%         当前（绿色）   记忆碎片、记忆残渣             记忆碎片 / 记忆残渣
  -------------------------------------------------------------------------

**5.2.2 变体一：危险资源区（Dangerous Resource Zone）**

**设计定位**：高风险高回报的资源采集区域。玩家面临"贪婪 vs 谨慎"的决策压力——资源翻倍但敌人更强，是否值得冒险？

| **增援规则** | 下一精英感知距离=走廊1/3 | 当前精英生命<30%时，下一个精英进入"警戒"状态（移速+20%，但尚未进入战斗） |

**5.2.6 多周目变体生成算法伪代码**

```pseudocode
// 文件：Assets/Scripts/Map/IntegratedMapSystem.cs
// 在GenerateGridMap()中，区块类型分配完成后调用

FUNCTION ApplyCycleVariants(blocks: Block[], currentCycle: int):
    // 统计各变体已生成数量
    variantCounts = { "DangerousResource": 0, "EliteChainCorridor": 0, "SpaceTimeDistortion": 0 }
    maxPerMap = { "DangerousResource": 2, "EliteChainCorridor": 1, "SpaceTimeDistortion": 1 }
    
    FOR EACH block IN blocks:
        // 安全区块不参与置换
        IF block.type == Safe: CONTINUE
        
        // 危险资源区：周目2+，置换资源区块
        IF currentCycle >= 2 AND block.type == Resource AND 
           variantCounts["DangerousResource"] < maxPerMap["DangerousResource"]:
            IF Random.value < 0.15:  // 15%置换概率
                block.variant = DangerousResource
                variantCounts["DangerousResource"]++
                ApplyDangerousResourceConfig(block)
        
        // 精英连战走廊：周目3+，置换战斗区块
        IF currentCycle >= 3 AND block.type == Battle AND 
           variantCounts["EliteChainCorridor"] < maxPerMap["EliteChainCorridor"]:
            IF Random.value < 0.10:  // 10%置换概率
                block.variant = EliteChainCorridor
                variantCounts["EliteChainCorridor"]++
                ApplyEliteChainCorridorConfig(block)
                // 确保前后区块为非战斗类型
                EnsureBufferBlocks(block)
        
        // 时空扭曲区：周目2+，置换任意非安全区块
        IF currentCycle >= 2 AND block.type != Safe AND 
           variantCounts["SpaceTimeDistortion"] < maxPerMap["SpaceTimeDistortion"]:
            IF Random.value < 0.08:  // 8%置换概率
                block.variant = SpaceTimeDistortion
                variantCounts["SpaceTimeDistortion"]++
                ApplySpaceTimeDistortionConfig(block)

FUNCTION EnsureBufferBlocks(variantBlock: Block):
    // 精英连战走廊前后保护
    prevBlock = GetPreviousBlock(variantBlock)
    nextBlock = GetNextBlock(variantBlock)
    
    IF prevBlock != null AND prevBlock.type == Battle:
        prevBlock.type = Resource  // 强制改为资源类型
    IF nextBlock != null AND nextBlock.type == Battle:
        nextBlock.type = Resource  // 强制改为资源类型
```

**6.4 实验室碎片区域多条件触发机制（P1-017 修复）**

原始设计中，实验室碎片区域仅当"智慧属性 > 30"时出现，导致纯物理流派（力量/敏捷）玩家在所有周目中无法体验该内容。这是Roguelite内容可见性设计的反模式——核心内容访问不应绑定于单一Build方向。

```

│   说明：原始条件保留，法术型Build最直接触达
│   适用：智慧加点流派
│
├─── 条件B ── 使用「智慧灵药」或「真理窥视药剂」后30分钟内 ────────
│   说明：炼金药剂作为临时通行证，物理流派可通过炼金系统自主解锁
│   适用：所有Build，消耗资源换取内容可见性
│   药剂效果：智慧灵药——临时提升智慧属性+20（持续30min）
│             真理窥视药剂——揭示隐藏路径+实验室碎片触发生效（持续30min）
│
├─── 条件C ── 当前周目 >= 3 且 智慧属性 >= 20 ─────────────────────
│   说明：多周目后门槛降低，老玩家已有充足Build优化空间
│   适用：周目3+玩家，属性能积累至此
│   设计意图：渐近性降低门槛而非一次性开放，保留门槛仪式感
│
└─── 条件D ── 共鸣属性 >= 25 ──────────────────────────────────────
    说明：利用已存在的"共鸣"属性作为替代鉴定条件
    适用：高共鸣通用Build（非智慧专精但走均衡路线）
    设计意图：共鸣属性在探索维度已有侦察价值（3.2节），
```

```
FUNCTION CanAccessLabFragment(player) -> bool:
```

```
- **沙暴机制**：精英#5和BOSS#10战斗中，沙暴周期性降低视野（但非完全遮挡），考验玩家对敌人AI模式的记忆
- **绿洲锚点**：营火#4是荒漠唯一安全区，玩家自然以此为"基地"向外辐射探索，形成轮辐式拓扑
- **双精英设计**：补偿荒漠区块数较少（8-10 vs 森林12），在更紧凑的空间内提供等量的挑战内容
```

**7.4 灵魂单局存储软上限机制**

-   **软上限值**：单局内灵魂持有量超过 **15,000** 后，后续获取灵魂量衰减为 **70%**
-   **硬性绝对上限**：单局灵魂持有量不得超过 **50,000**，超出部分直接流失
-   **UI提示**：达到软上限时，灵魂计数器旁显示黄色衰减图标；达到硬上限时显示红色满额图标
-   **设计意图**：
    -   防止玩家在安全区域无限刷取灵魂，破坏"资源紧张→决策取舍"的核心体验
    -   软上限提供预警，硬上限防止极端囤积
    -   配合"奢侈消费Sink"（洗点、武器特效重置等），引导玩家主动消耗而非无限积累
    -   数值锚定：15,000软上限约为50级角色升级总需求的9%，在保证成长空间的同时设置合理边界

    > **与经济系统协调说明**：软上限 15,000 与 [资源经济系统拆解](./幻宫_时空回响_资源与经济系统拆解.md) 中后期预期灵魂获取量（15,000-20,000）在数值上自然交汇。玩家在后期阶段将触及软上限，触发"囤积 vs 消费"的决策压力。此参数与 Source-Pool-Sink 模型中的奢侈消费 Sink（洗点、武器特效重置等）协同运作，形成完整的资源管理闭环。硬上限 50,000 在终局阶段（Boss/精英密集）提供绝对安全边界。

**九、 精简后的设计优势总结**

1.  **聚焦核心体验**：玩家时间更集中于战斗、资源管理和炼金合成循环。

2.  **降低认知负担**：无需记忆事件触发条件与结果，探索决策更直接（战或避、搜或不搜）。

3.  **提升节奏感**：移除事件对话与选择环节，使单局流程更紧凑，符合Roguelite的"短平快"节奏。

4.  **减少开发复杂度**：无需设计、编写、测试大量分支事件脚本，适合3-5人小团队。

5.  **强化系统深度**：将原本用于事件的开发资源，可用于深化战斗AI、炼金组合或地图生成算法。

**精简版地图系统设计愿景**：

幻宫是一片纯粹由战斗、资源与时空裂隙构成的领域。这里没有偶然的奇遇，只有通过观察、战斗与炼金才能揭示的规律与真相。每一次探索都是对玩家策略与决心的直接考验。

**10.1 地图区块生成（IntegratedMapSystem.GenerateGridMap）**

```pseudocode
// 文件：Assets/Scripts/Systems/IntegratedMapSystem.cs
// 实现路径：已有5×5网格框架，需按区块类型概率填充

FUNCTION GenerateGridMap():
    ClearOldMaps()
    gridMaps = new GameObject[gridSize, gridSize]
    
    // 1. 确定安全区位置（营火）
    safeRoomPos = (gridSize/2, gridSize/2)
    gridMaps[safeRoomPos.x, safeRoomPos.y] = Instantiate(safeRoomPrefab)
    
    // 2. 确定BOSS房间位置（边缘随机）
    bossRoomPos = GetBossRoomPosition()  // 不与安全区相邻
    gridMaps[bossRoomPos.x, bossRoomPos.y] = Instantiate(bossRoomPrefab)
    
    // 3. 填充剩余格子：50%资源 / 40%战斗 / 10%安全
    FOR x = 0 TO gridSize-1:
        FOR y = 0 TO gridSize-1:
            IF (x,y) is safeRoom or bossRoom: CONTINUE
            
            roll = Random.value
            IF roll < 0.50:          prefab = GetRandomResourcePrefab()
            ELSE IF roll < 0.90:     prefab = GetRandomBattlePrefab()
            ELSE:                    prefab = GetRandomSafePrefab()
            
            gridMaps[x,y] = Instantiate(prefab)
    
    // 3.5. 可达性验证（BFS广度优先搜索）
    // 从安全区出发，验证是否存在至少1条路径到达BOSS房间
    IF NOT ValidateReachability(safeRoomPos, bossRoomPos):
        // 策略A：使用最小生成树补全断开的连通分量
        ConnectDisconnectedComponents(safeRoomPos, bossRoomPos)
        // 策略B：若补全后仍不连通，重新生成整个网格
        IF NOT ValidateReachability(safeRoomPos, bossRoomPos):
            Debug.LogWarning("[MapGen] 可达性补全失败，重新生成...")
            ClearOldMaps()
            GenerateGridMap()  // 递归重试（最多3次，超过则使用强制直连）
            RETURN
    
    // 4. 生成时空裂隙（5%基础概率 + 负担修正）
    GenerateTimeRifts()

// ★ 可达性验证算法（BFS）
FUNCTION ValidateReachability(start: Vector2Int, target: Vector2Int) -> bool:
    visited = new bool[gridSize, gridSize]
    queue = new Queue<Vector2Int>()
    queue.Enqueue(start)
    visited[start.x, start.y] = true
    
    WHILE queue.Count > 0:
        current = queue.Dequeue()
        IF current == target: RETURN true  // 找到路径
        
        // 四方向探索（上/下/左/右）
        FOR EACH dir IN [(0,1), (0,-1), (1,0), (-1,0)]:
            neighbor = new Vector2Int(current.x + dir.x, current.y + dir.y)
            IF neighbor.x >= 0 AND neighbor.x < gridSize AND
               neighbor.y >= 0 AND neighbor.y < gridSize AND
               NOT visited[neighbor.x, neighbor.y] AND
               IsRoomTraversable(neighbor):  // 检查是否为可行走区块
                visited[neighbor.x, neighbor.y] = true
                queue.Enqueue(neighbor)
    
    RETURN false  // 不连通

// ★ 最小生成树补全（Kruskal算法简化版）
FUNCTION ConnectDisconnectedComponents(start: Vector2Int, target: Vector2Int):
    // 找到所有连通分量
    components = FindAllConnectedComponents()
    // 找到包含起点的分量和包含目标的分量
    startComponent = FindComponentContaining(start)
    targetComponent = FindComponentContaining(target)
    // 在两个分量之间找到最近的可行走节点对，添加连接路径
    AddBridgeBetweenComponents(startComponent, targetComponent)
```
