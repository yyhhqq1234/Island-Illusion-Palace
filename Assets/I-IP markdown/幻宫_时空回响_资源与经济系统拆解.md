**三、 资源消耗系统**

**3.0 Source-Pool-Sink 模型概览**

| 环节 | 内容 | 机制 |
|------|------|------|
| **Source（产出）** | 杀敌、采集、任务、分解 | 战斗掉落 + 地图采集 + 任务奖励 |
| **Pool（存储）** | 灵魂(软上限15,000/硬上限50,000)、灵魂精华(999)、材料(堆叠99/10) | 各资源独立上限，溢出流失或无法拾取 |
| **Sink（消耗）** | 角色升级、武器强化、召唤突破、炼金合成、负担恢复 | 见下方详细分配 |
| **奢侈Sink** | 洗点、武器特效重置 | 非必需但高价值消耗，吸收经济余量 |

**3.0.1 奢侈消费 Sink（经济余量吸收）**

| 消费项 | 消耗 | 限制 | 设计意图 |
|--------|------|------|---------|
| **属性洗点** | 500灵魂/次 | 仅在营火处可用，每局最多3次 | 允许Build纠错但成本显著，防止随意切换 |
| **武器特效重置** | 200灵魂 + 1稀有材料 | 重置武器+3/+5特效槽的随机特效，结果重新随机 | 为追求最优Build提供出口，消耗经济余量 |
| **召唤物词条刷新** | 300灵魂 + 2稀有材料 | 刷新永久召唤物的随机词条（品质不变），每召唤物每局限1次 | 终局追求项，吸收大量资源 |

-   奢侈Sink与核心Sink的消耗比例约为 **1:4**（奢侈消费约占灵魂总消耗的20%）
-   设计意图：确保核心成长路径（升级/强化）不被奢侈消费挤压，同时为富裕玩家提供资源出口，维持经济循环活力

**4.5 灵魂炼金：灵魂到灵魂精华的转换桥梁**

灵魂炼金是连接「灵魂（通用货币）」与「灵魂精华（稀有货币）」的确定性经济桥梁，解决玩家因精华掉落 RNG 不利而无法推进召唤物突破/武器强化的卡关风险。

**4.5.1 核心参数**

| 参数 | 数值 | 说明 |
|------|------|------|
| **兑换比例** | 灵魂 × 50 → 灵魂精华 × 1 | 锚定于单局灵魂总获取量（80,000-100,000），50:1 确保兑换成本显著 |
| **单局限次** | 10 次 | 折合 500 灵魂，约占单局灵魂总获取的 0.5%-0.6%，防止灵魂炼金替代核心掉落成为精华主来源 |
| **执行地点** | 营火处（炼金台） | 与材料分解、属性洗点等奢侈 Sink 同属营火决策点，强化"回营决策"的策略深度 |
| **解锁条件** | 无前置条件 | 新玩家即可使用，降低早期精华短缺的挫败感 |

**4.5.2 设计推导**

```
灵魂精华单局总获取量（核心掉落）：约 100-150（见 4.1 节）
灵魂炼金贡献上限：10 精华（10 次 × 50 灵魂）
灵魂炼金占比上限：10 / 100 = 10%（最少情况），10 / 150 ≈ 6.7%（充足情况）

标准 Build 精华消耗（见 4.2 节）：90 精华
核心掉落覆盖消耗：100-150 > 90，正常情况下无需灵魂炼金即可满足需求
灵魂炼金定位：补足机制——仅在精华掉落 RNG 极差时使用，而非常规获取手段
```

**4.5.3 使用场景分析**

| 场景 | 是否使用灵魂炼金 | 原因 |
|------|-----------------|------|
| 精华掉落正常（100-150） | 否 | 核心掉落已覆盖消耗，灵魂优先用于升级/武器强化 |
| 精华掉落偏低（60-90） | 部分使用（3-5次） | 补充缺口至 90，消耗 150-250 灵魂 |
| 精华掉落极差（<60） | 满额使用（10次） | 最大化补充，但仍无法完全弥补缺口（10精华 < 30缺口），需调整 Build 策略 |
| 召唤师 Build（精华消耗 60+） | 按需使用 | 召唤师精华需求高，灵魂炼金提供了不依赖 RNG 的确定性路径 |

**4.5.4 与经济系统的协同**

- **与灵魂软上限（15,000）的交互**：灵魂炼金消耗灵魂，延缓玩家触及软上限，间接鼓励"消费而非囤积"
- **与奢侈 Sink 的竞争**：灵魂炼金（500 灵魂满额）与属性洗点（500 灵魂/次）、武器特效重置（200 灵魂/次）共享灵魂资源池，形成"精华补足 vs Build 优化"的决策张力
- **与材料合成的区别**：材料合成（炼金台自由炼金）可通过材料组合产出灵魂精华（价值 25 匹配），但依赖材料库存且存在成功率风险；灵魂炼金为确定性兑换，无成功率风险但成本更高（50 vs 25 等效价值），形成"稳定高价 vs 风险低价"的双路径选择

**七、 防通货膨胀与防刷机制**

**7.1 资源获取限制**

-   **区域刷新**：敌人不重复刷新，清空区域后需前往新区域

-   **采集点限制**：单局内采集点不刷新

-   **精英/BOSS重生**：仅记忆守护者可重复挑战（为碎片），其他不重生

**7.2 惩罚机制**

-   **高负担状态**：\>70负担时，杀敌灵魂获取-30%

-   **死亡惩罚**：损失当前灵魂的20%（最低100）

-   **时空裂隙风险**：高风险高回报，可能进入无法应对的区域

**7.3 资源上限控制**

-   **灵魂**：单局软上限 **15,000**（超出后获取效率×0.7），硬上限 **50,000**（超出部分直接流失，无法拾取）。鼓励持续消费而非囤积；软上限 15,000 锚定于后期预期获取量（15,000-20,000），在玩家进入终局时形成自然的"消费还是囤积"决策压力。
-   **灵魂精华**：上限999（防止过度囤积）

-   **材料堆叠**：基础/稀有99，史诗/传奇10（鼓励使用而非囤积）

-   **灵魂之核槽位**：最大15（智慧25点+基础10），限制携带数量

**8.2 玩家体验目标**

-   **挫败感控制**：单局内至少能完成1-2个核心成长目标（如武器+3或主力召唤物升级）

-   **成就感节奏**：每15-20分钟应有明显的资源获取反馈

-   **多周目动力**：真结局需2-3周目合理规划，而非单纯刷资源

-   **流派平衡**：四种主流Build的单局资源需求差异不超过25%

**经济系统设计目标**：

构建一个让每一份资源都有明确去向、每一次战斗都有合理回报、每一次选择都有机会成本的清晰经济循环。玩家应始终在"现在使用"与"存到更划算时"之间做出有意义的策略抉择，而非简单囤积或随意挥霍。

FUNCTION CollectResource(type: ResourceType, amount: int, target: string):
    SWITCH type:
        CASE Soul:
            currentSouls += amount
            Debug.Log($"获得灵魂 ×{amount}（来源：{target}）")
        CASE SoulEssence:
            currentSoulEssence = MIN(currentSoulEssence + amount, 999)
            Debug.Log($"获得灵魂精华 ×{amount}")
        CASE Material:
            AddMaterial(materialType, amount)
        CASE SoulCore:
            AddSoulCore(enemyType, quality)
        CASE MemoryFragment:
            memoryFragmentSystem.CollectFragment(fragmentType)

**9.2 经验-灵魂消耗计算**

```pseudocode
// 文件：Assets/Scripts/Core/CharacterStats.cs
// 实现路径：需创建经验表

// ★ 不等距经验表生成公式
// Lv1-10:  100 + level × 20
// Lv11-20: 300 + level × 30
// Lv21-30: 500 + level × 40
// Lv31-40: 800 + level × 50
// Lv41-50: 1200 + level × 60

FUNCTION GetExpRequirement(level: int) -> int:
    IF level <= 10: RETURN 100 + level * 20
    IF level <= 20: RETURN 300 + level * 30
    IF level <= 30: RETURN 500 + level * 40
    IF level <= 40: RETURN 800 + level * 50
    RETURN 1200 + level * 60

// 累计至Lv50约160,000灵魂
```

**9.3 采集点交互（MaterialNode采集逻辑）**

```pseudocode
// 文件：新建 Assets/Scripts/Systems/MaterialNode.cs
// 实现路径：新建交互式采集点组件

public class MaterialNode : MonoBehaviour, IInteractable:
    public MaterialTypeEnum materialType
    public int quantity = 1
    public float collectTime = 2.0  // 采集耗时
    
    FUNCTION Interact(player: GameObject):
        // ★ 单局内不刷新，采集后消失
        IF isCollected: RETURN
        
        // 采集动画耗时
        StartCoroutine(CollectWithDelay(player))
    
    IEnumerator CollectWithDelay(player):
        yield return WaitForSeconds(collectTime)
        
        inventory = player.FindComponent<InventorySystem>()
        inventory.AddMaterial(materialType, quantity)
        
        isCollected = true
        // 可播放消失动画后Destroy
        Destroy(gameObject, 0.5)
```
